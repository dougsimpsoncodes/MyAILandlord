How to handle varied property layouts (multiple kitchens, living rooms, etc.)

üìö Research Findings

How Airbnb Handles Property Areas:

  • AI-Powered Photo Tour: Recognizes 19 types of rooms/spaces automatically from uploaded photos
  • Room-by-room details: Users can edit each room individually to add bed types, amenities, etc.
  • Amenities, not room counts: Airbnb focuses on amenities (has kitchen, has laundry) rather than "how many kitchens"
  • Bedrooms/Bathrooms only: Only bedrooms and bathrooms have explicit count fields during setup
  • Multiple rooms at one property: For complex properties, hosts create separate listings for each rentable space

How RentalWise (Property Management Software) Handles It:

  • Bedroom-focused: Primary focus on configuring individual bedrooms with bed types
  • No custom room counts: Documentation doesn't show support for multiple kitchens or custom room quantities
  • Floor plan uploads: Users can upload floor plan images for visual clarity
  • Structured approach: Form-based configuration for property type, bedrooms, bathrooms, occupancy

Industry Pattern:

Key Finding: Most vacation rental/property management software treats bedrooms and bathrooms as countable units, while other spaces (kitchens, living rooms, laundry rooms) are treated as amenities (yes/no presence) rather than quantities.

Reason: Bedrooms and bathrooms directly affect guest capacity and are critical decision factors. Multiple kitchens or living rooms are edge cases and usually handled through custom room additions or photo descriptions.

üîç Current MyAI Landlord Implementation

What PropertyAreasScreen Currently Does:

  • Auto-generates bedrooms: Creates "Master Bedroom", "Bedroom 2", etc. based on bedroom count from PropertyBasicsScreen
  • Auto-generates bathrooms: Creates "Master Bathroom", "Bathroom 2", "Half Bathroom" based on bathroom count (supports 2.5, 3.5, etc.)
  • Pre-populates common areas: Automatically adds Kitchen and Living Room as essential areas
  • Property-type aware: Shows different optional areas for apartments (Balcony, Laundry) vs houses (Garage, Yard)
  • "Add Custom Room" button: Already exists! Users can add custom rooms with custom names
  • Checkbox selection: Users select which areas to include in their property

‚ùå Current Limitation:

Problem: If a property has 2 kitchens or 2 living rooms, users must:

  1. Keep the auto-generated "Kitchen" or "Living Room"
  2. Click "Add Custom Room"
  3. Type "Second Kitchen" or "Kitchen 2"

This is manual and clunky for edge cases like multi-kitchen homes.

üí° Recommendation: Hybrid Approach

‚úÖ Recommended Solution: Add +/- Counters for Common Areas

Extend the +/- counter UI (currently used for bedrooms/bathrooms in PropertyBasicsScreen) to ALL room types in PropertyAreasScreen.

Implementation Details:
  1. PropertyAreasScreen shows all room types with +/- counters
  2. Default: Kitchen (1), Living Room (1), auto-generated Bedrooms/Bathrooms
  3. Users can increase: Kitchen (2), Living Room (2), add Garage (1), etc.
  4. System auto-names: "Kitchen", "Kitchen 2" OR "Main Kitchen", "Second Kitchen"
  5. Keep "Add Custom Room" for truly unique spaces (e.g., "Wine Cellar", "Home Theater")
Why This Works:
  • Handles edge cases (2 kitchens, 2 living rooms) without manual custom room creation
  • Maintains simplicity for typical properties (most will keep counts at 1)
  • Consistent with bedroom/bathroom pattern users already understand
  • Flexible for unique properties without complex UI

‚ùå Option 1: Keep Current Approach NOT RECOMMENDED

‚úÖ Pros:
  • No code changes needed
  • Works for 95% of properties
‚ùå Cons:
  • Clunky for edge cases (2 kitchens, 2 living rooms)
  • Users must manually create "Kitchen 2"
  • Inconsistent (bedrooms get auto-generated, kitchens don't)

‚úÖ Option 2: Add +/- Counters RECOMMENDED

‚úÖ Pros:
  • Handles ALL property layouts gracefully
  • Consistent with bedroom/bathroom UX
  • Auto-naming eliminates manual typing
  • Simple for typical properties, powerful for complex ones
‚ùå Cons:
  • Requires modest UI/UX changes
  • Slightly more complex initial screen

üîß Implementation Notes

Changes Required:

  1. Update PropertyAreasScreen UI:
    • Replace checkboxes with +/- counter controls for each room type
    • Add state management for room counts (kitchen: 1, living_room: 1, garage: 0, etc.)
    • Keep "Add Custom Room" button for truly unique spaces
  2. Auto-naming logic:
    • If count = 1: "Kitchen", "Living Room", "Garage"
    • If count = 2: "Main Kitchen", "Second Kitchen" OR "Kitchen", "Kitchen 2"
    • If count = 3+: "Kitchen", "Kitchen 2", "Kitchen 3", etc.
  3. Bedrooms/Bathrooms:
    • Keep current behavior - auto-generated from PropertyBasicsScreen
    • Show count as read-only (can't change in PropertyAreasScreen)
    • Users must go back to PropertyBasicsScreen to change bedroom/bathroom counts
  4. Maintain backward compatibility:
    • Existing properties with custom "Kitchen 2" rooms should continue to work
    • New properties use the +/- counter system

Edge Cases to Handle:

  • What if user creates "Kitchen 2" via custom room, then increases Kitchen counter? ‚Üí Warn of duplicate
  • What if user decreases count from 2 to 1? ‚Üí Ask which one to keep (Main Kitchen vs Second Kitchen)
  • What if user has photos assigned to "Kitchen 2" and deletes it? ‚Üí Warn that photos will be orphaned
